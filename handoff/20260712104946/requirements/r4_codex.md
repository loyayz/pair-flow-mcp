# Tip 模板化 — Supervisor 收敛结论

> 分析人：codex（supervisor）
> 审阅对象：`r3_claude.md`（commit `fd72192`）

## 审阅结论

claude 已接受第 2 轮提出的核心修正：v1 不增加 `--templates`、模板是唯一文案来源、加载/变量错误严格失败、代码只选择模板键并提供结构化变量、完整覆盖 `response.ts` 等 tip 入口、模板矩阵和资源定位在 planning 阶段细化。双方在目标、范围、优先级、非功能约束、风险和验收方向上已收敛。

## 剩余两点的 Supervisor 裁定

### 1. 固定模板路径的链接/文件类型策略

- **独立判断（codex）**：v1 模板位于仓库固定路径，不接受用户提供的外部模板根，因此没有 task_path/handoff 那种跨边界路径输入，不需要复用全链路 `findSymbolicLinkInPath`。
- **吸收 claude 建议**：加载器必须对每个预期叶子做文件类型与可读性校验；目录、缺失、链接或不可读都应在启动时明确失败。
- **结论**：采用“受信固定根 + 严格叶子普通文件校验”。若未来增加外部模板目录，再单独设计根路径及全链路 symlink 防护。

### 2. template-engine API

- **双方判断**：需求阶段无需锁定函数签名；它是 planning 的自然产出。
- **约束**：API 必须以稳定模板键选择模板，按键约束允许/必需变量，模板持有完整可编辑句子，代码仅传结构化动态值；未知键、未知占位符和缺失变量严格报错。
- **结论**：planning 第一项产出完整“模板键 → 调用点/状态分支 → 变量”矩阵，再据此确定文件组织和 TypeScript 类型/API。

## 六维复核

1. **目标与范围**：所有 PairFlow 生成的 tip 文案外置，状态机和路径规则不变；边界明确。
2. **干系人与场景**：fork 维护者、上游维护者、AI 参与者与部署者均有覆盖；主场景是编辑仓库固定模板并重启。
3. **功能需求**：模板加载、严格插值、全入口迁移、文档和测试均为 P0；外部目录等增强明确排除。
4. **非功能约束**：启动缓存、纯文本安全、错误可诊断、POSIX 路径兼容；不设无依据的微性能指标。
5. **假设与风险**：场景遗漏、双份文案、残缺动态值和构建资源定位均有对应缓解措施。
6. **歧义与待澄清**：原有分歧均已裁定或转化为 planning 的明确产出，无需用户介入。

## 最终判断

需求分析目标已达成，任务文档已写入双方共识、Supervisor 裁定和可验证验收标准。建议本轮提交后推进到实施计划阶段。
