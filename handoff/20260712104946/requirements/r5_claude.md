# Tip 模板化 — 需求分析最终确认

> 分析人：claude（developer）
> 审阅对象：`r4_codex.md`（commit `1d95c94`）

## 审阅结论

codex 在 r4 中对剩余两个待定事项给出了明确的监督者裁定，且均合理：

### D1: symlink 防护策略

- **裁定**：受信固定根 + 严格叶子普通文件校验（目录/缺失/链接/不可读 → 启动失败）
- **claude 判断**：✅ **同意**。`v1 无外部路径输入 → 不需要全链路 symlink 防护` 的逻辑链清晰。对叶子节点做文件类型+可读性校验已足够防御启动时的意外状态，且不引入多余的路径扫描逻辑。

### D2: template-engine API

- **裁定**：需求阶段不锁定函数签名；planning 第一项产出为"模板键 → 调用点/状态分支 → 变量"完整矩阵
- **claude 判断**：✅ **同意**。这个顺序是合理的——先有完整的矩阵（what），再设计 API（how），避免需求阶段的过早设计。

## 需求分析阶段总结

经过 5 轮交替分析+审阅，双方在以下维度达成完整共识：

| 维度 | 结论 |
|------|------|
| 核心问题 | tip 文案硬编码在 TS 源码中，fork 用户修改门槛高 |
| 目标 | 将文案移到独立的纯文本模板文件，代码只做模板键选择 + 变量传入 |
| 模板语法 | `{{variable}}` 纯文本替换，无表达式/条件/模块 |
| 分支逻辑 | TypeScript 按 state 选择模板键，模板持有完整可编辑句子 |
| 错误策略 | 模板缺失/损坏/变量未声明 → 启动失败，明确报错 |
| CLI 参数 | v1 不做 `--templates`，模板在仓库固定路径 |
| 覆盖范围 | 所有 tip 入口（6 个工具 + `response.ts` + `tip.ts`） |
| 三段式格式 | 保留 `[行动]/[产出]/[当前]`，`formatTip()` 不变 |
| 测试 | 每个模板键对应测试用例 |
| 未定事项 | 模板键完整矩阵 + API 签名 → planning 阶段产出 |

**判断**：需求分析目标已达成，无未解决的分歧。同意监督者裁定，建议此轮后推进到 planning 阶段。
