# Tip 模板化需求分析与第 1 轮审阅

> 分析人：codex（supervisor）
> 审阅对象：`r1_claude.md`（commit `b6eb683`）
> 权威规格：`docs/design.md`

## 1. 目标与范围

### codex 的独立判断

核心问题是 tip 文案与状态机代码耦合：fork 维护者为了改变 AI 行动指引，必须修改多个 TypeScript 文件，并承担逻辑误改、测试更新和上游合并冲突。目标应是建立“代码选择场景并提供变量，模板拥有文案”的单向边界。

P0 必须覆盖所有实际产生 tip 的入口，而不只 `tip.ts`：`register`、`confirm_task`、`advance`、`wait_for_turn`、`get_state`、`submit`、`response.ts` 统一业务错误，以及 `tip.ts` 的所有阶段/轮次分支。状态机、权限、轮次、路径与职责计算不在本需求中改变。

### 与 claude 对比

- 一致：文案外置、简单插值、状态分支留在代码、三段式语义不变、v1 不做条件语法/热加载/多语言。
- 补充：claude 的清单遗漏了 `response.ts` 统一业务拒绝 tip，且“所有接口”需要一份可核对的完整模板键清单。
- 分歧：claude 提出的 `--templates`、按 workflow 切换、多版本目录不属于原始目标。fork 用户直接修改仓库内默认模板即可解决问题，v1 不应扩张 CLI 与配置面。

## 2. 干系人与场景

### codex 的独立判断

1. **fork 维护者**：编辑仓库内模板，定制语气、语言和组织术语；最关心无需碰业务代码、变更可审阅。
2. **PairFlow 维护者**：新增状态分支时必须同时新增模板键和变量契约；最关心缺漏能被测试或启动校验发现。
3. **AI 参与者**：依赖 tip 执行文件、Git 和 MCP 操作；最关心动态路径、commit、身份和轮次不丢失。
4. **部署者**：构建/运行源码与未来可能的打包产物；最关心模板可可靠定位、缺失时错误明确。

主场景是 fork 后直接编辑默认模板并重启服务；回归场景是未编辑模板时既有测试与行为保持一致；维护场景是新增 tip 分支时缺少模板或变量立即暴露。

### 与 claude 对比

- 一致：fork 维护者、AI 参与者、上游维护者是主要干系人。
- 质疑：claude 将“复制到自定义路径并通过 CLI 指定”作为主场景，是尚未得到任务支持的过度假设。
- 补充：部署/打包模板定位是实现约束，但不等于必须对外增加路径参数。

## 3. 功能需求与优先级

### codex 的独立判断

**P0**

1. 建立稳定的模板键目录，逐一映射当前所有 tip 场景。
2. 提供纯文本模板加载器和 `{{name}}` 插值器；每个模板声明或由代码约束其必需变量。
3. 启动时加载默认模板并缓存，正常请求只做内存查找与字符串渲染。
4. 未知模板键、模板文件缺失/不可读、未知占位符、必需变量缺失均给出可定位错误，不得静默生成残缺指引。
5. 将代码中的人类可编辑文案全部迁出；代码只保留场景选择、结构化动态值和状态规则。
6. 保持 `docs/design.md` 规定的 `[行动]/[产出]/[当前]` 顺序与路径格式。
7. 迁移现有测试并增加加载、插值、校验及主要状态分支覆盖。
8. 在 README 或模板目录说明中记录如何编辑、变量含义及重启生效。

**P1（非本轮必要）**

- 外部模板目录、CLI 参数、overlay/fallback、多语言或模板版本化。

### 与 claude 对比

- 一致：纯文本双花括号插值、启动加载缓存、默认行为一致、测试覆盖。
- 修正：claude 的 `{{action}}/{{product}}/{{current}}` 若仍由 TypeScript 生成完整句子，会让主要 tip 文案继续硬编码；模板应拥有句子，代码只传结构化变量。
- 修正：所有模板共享宽泛变量表会掩盖缺失值。应按模板键校验变量集合。
- 分歧：不接受“模板缺失回退到代码常量”，因为它制造双份权威来源并违背任务目标。

## 4. 非功能约束

### codex 的独立判断

- **正确性**：默认模板渲染后的语义与现状一致；动态路径、identity、角色、round、phase、commit 不得丢失。
- **可诊断性**：启动/渲染错误包含模板键、文件路径和变量名。
- **性能**：启动加载、请求期内存渲染；无需未经基准支持的 `<1ms` 硬指标。
- **安全**：只做文本替换，不执行表达式、模块或命令；变量值不参与二次模板解析。
- **兼容性**：Node.js/TypeScript 版本遵循现有项目；Windows 路径输出继续转 POSIX 斜杠。
- **可维护性**：模板键、文件、变量契约和测试一一对应，新增场景不能无模板通过。

### 与 claude 对比

- 一致：UTF-8、缓存、纯文本替换、默认语义一致。
- 质疑：`<1ms` 和 `<50KB` 没有业务依据，作为设计倾向即可，不应成为验收硬门槛。
- 分歧：未知变量不能一处“保留原文”、另一处“替换为空”；两种都会隐藏错误。P0 应严格报错。
- 分歧：仓库内受信模板无须直接复用 task/handoff 的全路径 symlink 防护；若未来开放外部路径，再定义威胁模型。

## 5. 假设与风险

### codex 的独立判断

**假设**

- 状态分支继续由 TypeScript 决定，模板不需要逻辑语法。
- 修改模板后重启可接受。
- fork 维护者编辑仓库内默认模板即可满足当前目标。

**风险**

1. 场景遗漏：分散在多个工具中的 warning/error/success tip 未迁移。缓解：先生成完整调用点与模板键清单，并用扫描/测试门禁。
2. 代码仍生成完整句子：表面模板化但 fork 可编辑范围有限。缓解：模板持有可读句子，代码只传变量。
3. 双份默认文案漂移：文件与代码 fallback 不一致。缓解：模板是唯一文案来源，加载失败明确报错。
4. 动态值缺失导致危险指令：空路径或空 identity 被渲染。缓解：按模板严格校验必需变量。
5. 构建/运行路径差异导致找不到模板。缓解：planning 阶段明确源码运行与构建产物的资源定位策略并测试两者。
6. 过细文件粒度造成维护噪音，过粗又难定制。缓解：先列模板键，再按共同变量和编辑场景确定文件组织。

### 与 claude 对比

- 一致：简单插值足够作为 v1、模板与代码存在契约耦合、资源定位需要解决。
- 补充：遗漏场景、双份文案、空动态值和构建产物定位是更直接的交付风险。
- 不同：不把 fork 用户手工合并外部模板目录视为当前假设，因为外部模板目录本身尚未进入范围。

## 6. 歧义与待澄清

1. **是否支持 `--templates`**：claude 建议 v1 支持；codex 建议 v1 只支持仓库内默认模板，原因是原需求仅要求 fork 易修改，CLI 会扩大路径、发布和 fallback 设计面。
2. **缺失模板策略**：claude 建议回退到代码硬编码；codex 建议明确失败，原因是代码 fallback 与“所有 tip 从模板获取”矛盾。
3. **模板组织粒度**：claude 给出了阶段目录草案；codex 建议 planning 阶段依据完整模板键清单定稿，避免需求阶段锁死文件 API。
4. **未知/缺失变量策略**：claude 文中同时出现“保留原文”和“替换为空”；codex 建议统一严格报错，避免残缺行动指令。
5. **资源定位**：双方都认为需要解决 `tsx` 源码运行与构建产物；具体采用相对 `import.meta.url`、构建复制还是其他方式留给 planning 阶段验证。

## 结论

已将双方一致项、codex 的必要补充和明确分歧写入任务文档。当前不存在必须立即询问用户才能继续分析的阻塞项；上述四项分歧应在下一轮交替审阅中收敛，未收敛前不应进入实施计划。
