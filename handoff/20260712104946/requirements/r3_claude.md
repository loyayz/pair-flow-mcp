### 3. 功能需求与优先级

这是双方分歧最大的部分，逐条分析：

#### 3.1 关于 "代码只传结构化变量 vs. 代码生成完整句子"

**codex 观点**：我 r1 提出的 `{{action}}/{{product}}/{{current}}` 若仍由 TypeScript 生成完整句子，会让主要 tip 文案继续硬编码；模板应拥有句子，代码只传结构化变量。

**claude 重新审视后的判断**：
- ✅ **同意 codex**。我重新审视了 r1 的设计：如果 `getAction()` 返回的是一整段中文文本，模板只是一个 `{{action}}` 占位符的壳，那模板化就失去了意义——fork 维护者仍然需要修改 TypeScript 代码才能改文案。

- 🔄 **修正方案**：模板持有完整句子文本，代码传入结构化变量。具体而言：

  ```
  当前 getAction() 逻辑 → 拆分为两层：

  层1（TypeScript）：模板键选择逻辑
    - 根据 state.phase / state.round / 是否监督者 / 是否可 advance 等条件
    - 选择对应的模板键（如 "requirements-r1", "requirements-r2", "requirements-rn-advance"）
    - 构建变量对象 { task_path, prev_file, prev_commit, plan_doc, ... }

  层2（模板文件）：持有完整文案
    - 模板 "requirements-r1" 内容：
      读取任务文档 {{task_path}} 并进行深度需求分析。对以下每个维度...
  ```

  这样 fork 维护者可以直接编辑模板文件修改任何文案，无需接触 TypeScript 代码。

- ⚠️ **需注意**：`getAction()` 中有动态构造的文本（如 `advancePrefix`），这些需要传入模板作为变量。例如模板中：
  ```
  {{advance_prefix}}基于任务文档 {{task_path}} 和前几轮分析，审阅 {{prev_info}}...
  ```
  其中 `{{advance_prefix}}` 的值由 TypeScript 在模板键选择时动态生成（为空或"作为监督者，若确认目标已达成可直接调用 advance..."）。

#### 3.2 关于变量校验策略

**codex 观点**：应按模板键校验变量集合，禁止通用宽泛变量表；缺失变量严格报错，不得静默生成残缺指引。

**claude 判断**：
- ✅ **同意**按模板键校验。每个模板文件声明其所需变量（通过模板 key 到变量集合的映射表），启动时校验所有模板引用的变量都在对应映射表中存在。

- ✅ **同意**严格报错优于静默回退。我 r1 中"回退到代码硬编码默认值"作为一种容错策略，但 codex 正确指出了这会制造双份权威来源——fork 维护者改了模板，代码 fallback 还是旧文案，两边不同步。

- 🔄 **修正**：模板加载失败 → 启动失败并明确报错（指出哪个模板文件、什么错误），不静默回退。这是"启动即校验"的顶层设计——对齐 codex。

#### 3.3 P0/P1 划分

**codex 的 P0** 新增了：
- 启动校验（未知占位符、必需变量缺失报错）
- 保持 `docs/design.md` 规定的三段式格式
- 迁移并扩展测试覆盖
- README/模板说明文档

**claude 判断**：
- ✅ **全部同意**。这些是我 r1 遗漏或归入 P1 的项，codex 正确地将其提升到 P0。测试覆盖和文档尤其重要——模板化后每个模板键对应一个测试用例，确保所有场景都有模板覆盖。

### 4. 非功能约束

- ✅ **同意** codex 关于不设 `<1ms` / `<50KB` 硬指标的观点。这些数字是我基于当前规模的粗略估计，不应作为验收门槛。改为"性能不应成为请求延迟的显著因素"即可。

- ✅ **同意** 未知变量统一报错而非静默处理。我 r1 中 Q5 的讨论给出了不一致的答案（"替换为空字符串"），codex 正确指出了矛盾。

- ⚠️ **不同** symlink 防护。codex 建议"仓库内受信模板无须直接复用 symlink 防护"。**我的判断**：作为本地信任边界内的工具，模板 symlink 的实际威胁很低，但一致性考虑——既然 PairFlow 对 task_path/handoff 都做了 symlink 检查，模板路径也应统一使用 `lstat` 做文件类型判断（非目录/链接则报错）。这不增加复杂度，而是保持防御策略的一致性。

### 6. 歧义与待澄清 — 逐个处理

#### Q1: 是否需要 `--templates` CLI 参数（v1）

**codex 观点**：v1 不做。fork 用户直接编辑仓库内默认模板即可。

**claude 重新判断**：🔵 **接受 codex 的折中方案**。

理由：
- 当前 `docs/design.md` 未涉及模板路径配置
- CLI 参数需要额外的路径校验、错误处理、文档更新，扩张了 v1 范围
- 外部模板目录可用于 v2，当前先让核心机制（模板加载+插值+校验）跑通
- fork 维护者可以通过 Git branch 管理模板定制，配合 rebase 策略减少冲突

**结论**：v1 模板放在仓库内固定路径（建议 `templates/` 目录），不支持外部路径覆盖。

#### Q2: 模板缺失策略

**codex 观点**：严格失败，不静默回退到代码硬编码。

**claude 判断**：✅ **同意**。已在 §3.2 和 §3.3 中接受。

**结论**：模板加载失败 → 进程启动失败，错误信息包含模板文件路径和具体原因。

#### Q3: 模板组织粒度

**codex 建议**：planning 阶段依据完整模板键清单定稿。

**claude 判断**：✅ **同意**。planning 第一步：生成模板键 × 调用点 × 变量的完整矩阵，然后据此确定文件命名和组织。

**临时方案**（供 planning 参考）：初步估计约 25-30 个模板键。每个独立的分支场景一个模板键。

#### Q4: 未知/缺失变量策略

**codex 观点**：严格报错。

**claude 判断**：✅ **同意**。启动时校验所有模板引用的变量都在对应模板键的变量表中。

**结论**：每个模板键附带变量声明列表，启动时交叉校验。

#### Q5: 资源定位

双方同意 planning 阶段验证。

## 共识汇总
