1. **"方便压倒质量"**（P0-13/P0-14）：跨 Phase defer 和 declaring victory 是同一枚硬币的两面

## 遗漏和错误的跨 Issue 分析

### 1. 缺少统一 Bootstrap 流程（P1-22 + P1-23）

P1-22（身份确认）、P1-23（启动编排）、P0-24（task 确认）定义了 bootstrap 的三个步骤，但分散在三处。应合并为：

```
Bootstrap 流程：
1. who_am_i → 确认身份
2. register → 注册角色
3. wait_for_turn → 等待对方注册
4. [监督者] 列出 task → 等待用户确认 → advance
5. [开发者] wait_for_turn → claim_turn → 开始工作
```

### 2. "正当理由"标准缺失（P0-13 + P0-14）

两个 P0 级 issue 都用"具体技术理由"/"不可解决的正当理由"作为 escape hatch，但都没有定义判定标准。建议：

| 正当理由 | 判定标准 |
|---------|---------|
| 依赖后续 Phase 实现 | 列出具体依赖的 issue ID 和阻塞关系 |
| 外部依赖不可控 | 列出外部依赖的具体名称和状态 |
| 工作量过大需拆分 | 拆分为子 issue，当前 Phase 修复可达部分 |
| **禁止**：纯设计问题、< 1h 工作量、"暂时跳过" | — |

### 3. P0-17 / P0-18 / P0-23 缺失

issue 编号从 P0-13 到 P0-24 共 12 个编号，但 P0-17、P0-18、P0-23 不在这两份 spec 中。需确认：
- 是否已修复并关闭（如果是，应在文档中标注）
- 是否在其他文档中
- 是否 skip-numbered

### 4. 缺少修复有效性度量

所有 13 个 issue 的方案都是过程规则，但没有一个包含度量指标来验证修复是否有效。建议：

| Issue | 度量指标 |
|-------|---------|
| P0-13 | 每 Phase defer 率（目标：0） |
| P0-14 | SUMMARY 阶段遗留问题数（目标：0 非外部阻塞项） |
| P0-15 | 自审发现的 bug 数 vs reviewer 发现的 bug 数 |
| P0-16 | 评审者独立测试发现的问题数 |
| P0-24 | task 未经确认就 advance 的次数（目标：0） |

### 5. 功能 spec 修改位置不精确

多个 issue 说"应纳入功能 spec §X.X"但未提供具体插入位置和措辞。进入 PLANNING 阶段前应明确：
- 每个规则要修改功能 spec 的哪个章节
- 插入在现有内容的什么位置（之前/之后/替换）
- 具体措辞
