## 三、本轮新增问题

### P1-19: 规则修订编码（r18-N1 等）含义未定义

**定位**：§5.3（r18-N1、r36-N1、r40-N1、r46-N1）、§11（r32-N1）

**问题**：spec 中出现 5 个规则修订编码，共 12 次，但**spec 从未定义这些编码的含义**：

| 编码 | 出现位置 | 出现次数 | 含义（推测） |
|---|---|---|---|
| r18-N1 | §5.3 line 319 | 1 | 监督者全面通读义务 |
| r32-N1 | §11 line 665 | 1 | 规约分发机制 |
| r36-N1 | §5.3 line 347, 357, 912 | 3 | 阶段报告内容规范 |
| r40-N1 | §5.3 line 319, 321; §3 line 70, 80, 85; §11 line 686; §17 line 910 | 7 | advance 前全面通读清单 |
| r46-N1 | §5.3 line 378 | 1 | IMPLEMENTATION 多循环支持 |

**使用者困惑**（用户直接提问）："需求文档中有些规范（比如 347 行的 r36-N1）这个编码是什么？"

可能的解读：
- `r36` = 规则第 36 号？某个轮次的第 36 条？
- `N1` = 新规则 1？Note 1？版本 1？
- 这些编码与 §11 rules_catalog 的 `id` 字段什么关系？

**方案建议**：二选一——

**方案 A（保留编码 + 定义含义）**：§11 增加"规则修订编码定义"段落：
> 规则修订编码格式为 `r{序号}-N{版本}`，其中 `r{序号}` 是规则在 spec 演进过程中的引入序号（非 rules_catalog 的 id），`N{版本}` 是该规则的版本号（N1 = 首次引入，N2 = 修订一次，以此类推）。编码用于 spec 内部交叉引用（如"r40-N1 可验证性升级"），与 rules_catalog 的 `id` 字段独立——catalog id 是编码时的规则标识，r{序号}-N{版本} 是 spec 演进时的修订标识。

**方案 B（删除编码 + 纯章节引用）**：删除所有 `r{序号}-N{版本}` 编码，改为纯章节引用。例如：
- `（r36-N1）` → `（见本节"阶段报告内容规范"）`
- `（r40-N1）` → `（见本节"advance 前置条件"）`
- `（r46-N1）` → `（见本节"IMPLEMENTATION 多循环支持"）`

**倾向方案 B**：理由——
1. 编码体系增加维护负担（spec 修改时需同步更新编码版本号，与 §11 catalog 一致性维护重复）
2. 纯章节引用更直观，使用者无需查找编码定义
3. rules_catalog 的 `id` 字段已承担"规则标识"职责，spec 内部不需要第二套编码
4. 当前 5 个编码无统一分配规则（r18/r32/r36/r40/r46 序号不连续，无法推断分配逻辑）

**rationale**：用户直接提问证明这些编码对使用者不透明。spec 是权威设计文档，内部引用应自洽可读，不应依赖未定义的编码体系。
