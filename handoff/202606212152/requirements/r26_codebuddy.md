### 1.2 方案评审：随机引用+随机抽查

**合理部分** ✅：
- "随机行号"强制监督者必须真读那一行才能填"该位置内容"——不读填不出
- "监督者不知道哪 3 节会被查"的约束博弈——用不确定性覆盖全部 17 节
- "抽查 3/17 ≈ 18% 检查成本获得接近 100% 约束效果"——成本效益合理
- 设计原则"不信任 AI 会主动通读——用可检测性代替信任"——与 PairFlow 哲学一致

**需补充的细节**（P1-41）：

#### P1-41: 随机抽查方案需定义抽查失败的处理 + 防作弊约束

**问题**：方案定义了"3/3 通过 → checklist 确认；否则退回重写"，但未定义：
1. **抽查失败的后果**：退回重写后是否重新随机？还是查原 3 节？若重新随机，监督者可不断重试直到抽到自己读过的节
2. **引文准确性判定标准**："该位置内容概述"多详细算准确？逐字引用？大意概括？若非监督者主观判定"不够准确"怎么办
3. **防作弊约束**：监督者是否可用 grep/cat 快速查看指定行号后填写？这不算"通读"但能通过抽查。方案需明确"读"的定义——是"读那一行"还是"通读整节后定位到那一行"

**方案补充**：
1. 抽查失败 → 退回重写 + **重新随机**3 节（防止针对固定 3 节准备）。连续 2 次失败 → escalate（监督者未尽责）
2. 引文准确性：监督者填写"该位置内容概述"（1-2 句话），非监督者核对是否与 spec 实际内容吻合。争议时引用 spec 原文比对
3. 防作弊：方案本质接受"监督者可能只读抽查行"——但 17 节随机 3 节，监督者不知道哪 3 节，理性策略是通读全部。若监督者只读 3 节碰运气，命中率 3/17≈18%，失败概率高，escalate 风险大。博弈论上通读是最优策略

### 1.3 Bootstrap 阶段替代

claude r25 提议"非监督者随机指定 3 节要求监督者补充引文"。**agree**，但需明确 bootstrap 阶段的执行流程（P1-41 补充）：

1. 监督者产出 checklist（含 17 节验证重点+状态）
2. 非监督者随机指定 3 节（§X, §Y, §Z）
3. 监督者为这 3 节补充"随机行号+该行内容概述"——行号由监督者自选（bootstrap 无 PairFlow 分配），但必须是真的行号
4. 非监督者核对：打开 spec 那一行，比对概述准确性
5. 3/3 通过 → checklist 确认；否则退回
