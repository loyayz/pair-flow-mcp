# 设计一致性审计与回归基线

**状态**：待处理

`docs/design.md` 是 PairFlow 实现的唯一权威来源，但当前缺少一份系统证据，证明关键设计条款均已落实到代码并由测试覆盖。本任务对当前实现执行一次完整的一致性审计，修复发现的偏差，并建立必要的回归基线。

**设计规格**：`docs/design.md`

## 已确认方案

1. 本任务是一次性的设计一致性审计，不是新功能开发。（用户、codex）
2. 以 `docs/design.md` 为唯一判断依据，建立“设计条款 → 实现位置 → 测试证据”的核对清单。（用户、codex）
3. 发现实现与设计不一致时，默认修改代码和测试以符合设计，不得自行修改设计来迁就当前实现。（用户、codex）
4. 只补充证明设计行为所必需的回归测试，不建设通用运行时协议校验框架。（用户、codex）
5. 使用真实默认模板和真实调用变量覆盖关键行动语义，避免只验证测试 fixture 内部自洽。（codex）
6. 完成后，“代码必须符合设计文档”成为所有后续任务的常规验收原则，不再为每次开发重复创建审计任务。（codex）

## 审计方法

按照 `docs/design.md` 的章节和规范性条款逐项核对：

- 为每条关键规则记录对应实现文件、入口和状态分支；
- 记录现有自动化测试及其覆盖的正常、拒绝、并发、取消、恢复和边界场景；
- 将没有实现证据、没有测试证据或行为不一致的条目标记为缺口；
- 对缺口实施最小修复并补充直接回归测试；
- 运行 PairFlow 自身的测试、TypeScript 类型检查和构建，形成最终验收记录。

核对清单是审计证据，不成为 `docs/design.md` 的替代规格。发生歧义时应回到设计原文判断；若设计本身确实矛盾或无法实现，停止该条修复并提交明确问题，由用户裁定后再修改设计。

## 范围

### 必须覆盖（P0）

- register、confirm_task、get_state、wait_for_turn、submit、advance 的设计行为。
- phase、round、turn、sub_phase、角色与权限门禁。
- token、identity、workflow 和任务确认生命周期。
- 结构化 action、reason、required output、references 和 tip 的一致语义。
- 长轮询、latest-wins、取消、超时和 turn claim 的并发边界。
- handoff/meta、路径安全、原子持久化和崩溃恢复。
- 模板加载、变量契约，以及真实默认模板的关键行动语义。
- SUMMARY 完成、归档和工作流终止行为。
- `docs/design.md` 明确规定的错误和幂等行为。

### 本任务不做

- 不新增 `validateInstruction()` 或其他生产运行时校验层。
- 不设计新的协议版本、状态、工具、阶段或工作流分支。
- 不为了提高覆盖率而重构无关代码或引入通用框架。
- 不评价 requirements、planning、implementation 或 summary 的内容质量。
- 不修改设计来合理化已有实现偏差，除非用户先明确裁定设计变更。
- PairFlow 运行时不执行测试、构建、Git 或其他外部命令。

## 修复原则

- 修改应保持最小，只修复能够由设计条款直接证明的偏差。
- 测试应验证外部行为和关键状态，不复制实现细节。
- 已有测试与设计冲突时，以设计为准，同时修正实现和错误测试预期。
- 设计未规定的行为不得在本任务中自行扩展为新契约。
- 发现可能影响兼容性或现有归档的数据结构变更时，应单独记录并请求用户裁定。
- PairFlow 的任何组件均不得执行 Git 命令；开发者可在 PairFlow 之外使用 Git 管理本任务代码。

## 产出

- 一份完整的设计条款、实现位置和测试证据核对清单。
- 审计发现清单，包括已确认一致项、已修复偏差和等待裁定项。
- 对已确认偏差的最小代码修复。
- 覆盖修复行为及此前缺失关键设计分支的回归测试。
- PairFlow 自身测试、类型检查和构建的验收结果。

## 验收标准

- `docs/design.md` 的每条关键规范都有可定位的实现证据和测试证据，或被明确列为等待用户裁定的问题。
- 所有已确认的实现偏差均已修复，且有能在修复前失败、修复后通过的回归测试。
- 真实默认模板与真实调用变量的关键行动语义得到覆盖。
- PairFlow 自身的全量测试、TypeScript 类型检查和构建通过。
- 没有新增运行时协议校验框架、工作流能力或与审计无关的重构。
- 审计结果明确说明其依据是任务执行时的 `docs/design.md`，后续设计变更仍需相应更新实现和测试。
- 实现和验收不会使 PairFlow 运行时执行 Git、测试或构建命令。
