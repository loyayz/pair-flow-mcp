# 权威产物与最终交付清单

**状态**：待处理

PairFlow 工作流会产生多轮 requirements、planning、implementation 和 summary 产物。当前归档可以完整追溯每轮提交，但工作流结束后缺少稳定、机器可读的权威入口：客户端无法直接判断哪份计划被双方接受、下一阶段必须读取哪份产物，以及哪份 summary 是最终报告。

**设计规格**：`docs/design.md`

## 已确认方案

采用“全阶段权威产物 + 最终交付清单”方案：

1. 阶段推进表示双方已经接受本阶段结论；PairFlow 据此记录该阶段的权威产物路径、调用方声明的 commit hash 和批准方。（用户、codex）
2. 下一阶段的结构化 `references` 指向已接受的权威版本，而不是默认指向第一轮草稿或由客户端自行猜测。（codex）
3. 每个 reference 明确标记为必读或可选，Agent 不需要从 tip 文案推断依赖强度。（codex）
4. SUMMARY → IDLE 时返回机器可读的最终交付清单，包括最终 summary、各阶段权威产物、批准链和归档目录。（用户、codex）
5. 验证命令与结果只记录 Agent 主动上报的证据；PairFlow 不执行、不复验，也不判断证据内容是否真实或充分。（用户、codex）
6. PairFlow 不整理或改写 Git 历史，不自动创建提交、分支、PR 或远程交付。（用户、codex）

## 永久边界

**PairFlow 永远不会执行 Git 命令。**

这条边界适用于 Server、SDK、CLI 及任何后续 PairFlow 组件，包括但不限于：

- 不执行 `git status`、`git diff`、`git log`、`git rev-parse` 等只读命令；
- 不执行 add、commit、branch、merge、rebase、squash、push 或 PR 操作；
- 不通过 Git 库、子进程或其他方式隐式查询或修改仓库；
- 不验证调用方提交的 commit hash 是否存在、属于当前分支或对应某个文件；
- Git 相关字段均为 Agent 或用户提供的声明数据，PairFlow 只按协议记录和返回。

## 权威产物语义

- “权威”表示该产物已通过 PairFlow 状态机中的双方审阅和 Supervisor 推进，不表示 Server 判断了内容正确性。
- 每个已完成阶段至多有一个当前权威产物；发生修订并再次被接受时，新版本取代旧版本成为权威入口，旧记录仍保留在归档中。
- 权威产物至少包含 phase、file_path、提交者、接受者、接受时间，以及调用方声明的 commit hash（若该阶段契约要求提供）。
- implementation 可保留多轮 coding/review 追溯记录，同时明确最终被接受的实现提交与评审产物。
- 最终 summary 必须指向最新被明确接受的总结版本，而不是简单选择 summary 目录中的最大轮次或固定文件名。

## 最终交付清单

工作流完成响应至少提供：

- workflow 标识与归档目录；
- requirements、planning、implementation、summary 的权威产物；
- 最终 summary 的路径、声明 commit 和批准方；
- 各权威产物之间的必读/可选引用关系；
- Agent 主动上报的验证证据及其报告者；
- Server 可直接从工作流状态得出的可选统计摘要，例如阶段耗时、轮数和提交次数；
- 明确说明 Git 信息与验证证据均为调用方声明，未经 PairFlow 复验。

具体字段名与数据结构在实施设计中确定，但必须属于稳定的结构化协议，不得要求客户端解析自然语言 tip。

## 范围

### 必须覆盖（P0）

- 各阶段权威产物的确定、替换与持久化规则。
- 阶段转换时引用权威产物，并区分必读与可选 reference。
- SUMMARY 完成时确定最终被接受的 summary。
- 工作流完成响应中的结构化最终交付清单。
- 不引入独立指标系统的前提下，返回 Server 已知的轻量工作流统计。
- 崩溃恢复后权威产物和批准链的重建规则。
- 多轮修订、缺轮次归档和重复请求下的确定性行为。
- 权威记录、归档 meta 和完成响应之间的一致性测试。

### 本任务不做

- 不读取或评价产物正文是否正确、完整或高质量。
- 不执行测试、构建、静态检查或任何外部验证命令。
- 不执行任何 Git 命令，也不验证 commit hash。
- 不自动暂存、提交、建分支、合并、rebase、squash、push 或创建 PR。
- 不删除或改写历史 handoff 产物。
- 不根据工作区或 Git 状态决定工作流是否可以结束。
- 不建设 metrics 存储、dashboard、heartbeat、进度推断或独立可观测性系统。

## 质量与安全约束

- Server 状态机是“哪个产物已被流程接受”的唯一权威；客户端不得自行选择最终版本。
- PairFlow 只判断协议事件和状态转换，不把 Agent 声明当作已独立验证的事实。
- 路径、identity、commit 字符串和归档记录继续遵循 `docs/design.md` 的校验与安全边界。
- 重复 submit/advance 或响应重试不得产生多个互相冲突的权威版本。
- 老的非权威产物继续可追溯，但不得被后续结构化指引误标为必读的当前版本。

## 验收标准

- planning 等阶段经过多轮修订时，下一阶段只把最终被接受的版本作为权威必读引用。
- SUMMARY → IDLE 返回唯一、明确的最终 summary，以及各阶段权威产物和批准链。
- 客户端无需扫描归档目录或解析 tip 即可定位最终交付入口。
- 崩溃恢复后返回的权威产物与未崩溃时一致。
- Agent 上报的验证结果和 commit hash 被明确标记为声明数据，PairFlow 不对其真实性背书。
- 自动化测试证明 PairFlow 的该流程不会启动 Git、测试、构建或其他外部命令。
- 工作流结束不会自动修改 Git 历史或仓库状态。
