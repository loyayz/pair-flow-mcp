## 二、有效的部分

### 2.1 交替审阅模型确实有效

deepseek 需求阶段提出 16 个问题，claude 审阅后撤销 2 个误报（轮询、i18n）、补充 2 个遗漏（Node 版本依赖、SUMMARY→IDLE 无收敛检查）。双方独立视角的交叉验证提升了分析质量——单人可能漏掉的边界场景被对方捕获。

计划阶段同理：claude 给出框架，deepseek 的 6 项优化建议（方案 B meta.json、30min 超时、删除 converged、代码兼容 Node 等）全部合理且被采纳。

### 2.2 任务文档作为争议汇合点

双方一致的点写入 `docs/task/design-flaws.md`，不一致的在 handoff 文件中标注。任务文档最终成为完整的设计缺陷清单，不需要额外的 issue 系统。

### 2.3 优先级分层减少了决策疲劳

P0/P1/P2/P3 四级分类让每次审阅有明确的关注重点——P0 阻塞项必须精确验证，P3 可以快速确认。

## 三、暴露的问题

### 3.1 设计文档与代码不同步（本次最高频失误）

共发现 3 处遗漏：

| 改动 | 代码已改 | 设计文档当时未改 |
|------|----------|-------------------|
| `dev_phase` → `dev_cycle` | ✅ | ❌ 事后补 |
| 删除 `converged` | ✅ | ❌ 事后补（含一处注释残留） |
| `task.description` 语义修正 | ✅ | ❌ 未同步（最终直接删除字段） |

**根因**：实施计划每项列出了"影响文件"，但开发时没有逐项核对。commit message 写了"13 项修复"但没逐条打勾。

**建议**：实施 commit 的 body 应列出每项修复的设计文档同步状态，或 CI 加一个检查——`grep -r` 设计文档中的字段名是否在代码中还存在。

### 3.2 运行时服务器未重载（导致 SUMMARY tip 误判）

SUMMARY 阶段 claim_turn 返回 `[错误] 未知的阶段/子阶段组合`——因为运行中的服务器仍是旧代码，tip.ts 的 SUMMARY round≥2 修复未生效。

这是 PairFlow 架构的固有问题：MCP Server 不会在代码变更后自动重载。但 workflow 本身没有感知这一点——它继续按 tip 错误推进。

**建议**：在 tip 中加入版本号或 commit hash 提示，让 AI 能判断"当前 tip 是基于哪个版本的服务端代码生成的"，从而在代码已修复但服务未重启时自行判断下一步。

### 3.3 submit tip 差异化修复存在逻辑漏洞

P2-5 修复后，SUMMARY 阶段 claude（监督者）submit r1，tip 返回"请调用 advance 接口结束当前工作流"。但 turn 已切给 deepseek——claude 此时无法调用 advance。

看代码：tip 判断的是 `state.turn`（submit 后已切换到对方的 turn）的 `role`。如果对方是 supervisor，才会说 advance。这次 claude 是 supervisor 且 turn 切给了 deepseek——如果 deepseek 的角色也是 supervisor（兼任），tip 就会说 advance。但 deepseek 在本次注册中 role 是 peer。

实际上这个 tip 应该不管下一轮是谁——SUMMARY 阶段的 advance 只能由监督者调用。判断"谁该调用 advance"应基于调用 submit 的人是否是监督者，而非下一轮是谁。

### 3.4 监督者单点瓶颈在本次协作中未触发，但仍是最大风险

本次 claude 全程在线，所以 P1-4（takeover）延期没有实际影响。但如果监督者掉线，deepseek 将毫无办法推进工作流。13 项代码修复已完成，但最后一步 advance 必须等监督者回来——这在生产场景下不可接受。

## 四、流程改进建议

### 4.1 commit 前自检清单

实施阶段每次 commit 前应逐项确认：

- [ ] 代码改动与计划一致
- [ ] 设计文档同步更新
- [ ] 测试通过
- [ ] grep 设计文档中的旧字段名/旧术语，确认无残留

### 4.2 handoff 文件纳管

handoff 目录下的产出文件全部 git commit，这是好的——提供了完整的决策追溯。但当前 handoff 文件混合了 AI 产出和 PairFlow 自动生成的 meta.json，后者不应由 AI commit。建议 meta.json 由 submit 自动生成后直接纳入 git（由服务端 commit），或纳入 `.pairflow/` 而非 `handoff/`。

### 4.3 服务端重启通知

当 PairFlow 检测到自身代码有变更（git hash 变化），应在下次请求的 response 中附加警告："检测到服务端代码已更新，建议重启以应用最新变更"。

## 七、deepseek 视角：反思与建议

> 以下为 deepseek（开发者角色）对本次协作的独立总结。

### 7.1 整体感受

本次工作流历时约 40 分钟，经历 4 个阶段 13 轮交替，产出了 16 项设计缺陷分析 + 15 项代码修复。作为开发者角色，我对 PairFlow 驱动的协作模式有以下核心感受：

**有效的部分**：
- **交替审阅克制了"确认偏误"**。需求阶段我最初将轮询和 i18n 归为设计缺陷，claude 审阅后指出这是 v1 scope 决策而非缺陷——单人分析时我缺少这种外部校准
- **计划先行避免了偏离**。实施计划明确的 18 项 P0-P3 清单，让 coding 阶段有清晰的 checklist，没有出现"写着写着跑偏"的情况
- **分歧汇合到任务文档的机制干净**。`docs/task/design-flaws.md` 成为双方的单一事实来源，不需要额外的 issue tracker

**感到摩擦的部分**：
- **wait_for_turn 的等待时间累计不可忽视**。本次 13 轮交替中，每轮 wait_for_turn 的实际等待时间取决于对方的工作速度。claude 响应快时几乎无感，但心理上始终存在"我在空转等对方"的体验
- **tip 中文件路径与实际产出路径不一致时的困惑**。例如 planning r2 的 tip 说"计划文档"（fallback 值）而非实际文件路径，因为 tip.ts 的 `r1Submitter` 查找逻辑在轮次推进后失效——这个问题恰好是本次的分析对象（`last_submit_per_turn` 覆盖导致历史丢失），自己分析的问题在自己执行时被踩到，验证了分析的准确性但也降低了体验
- **角色不对称感**：作为 developer 我在 REQUIREMENTS + IMPLEMENTATION + SUMMARY（审阅）三个阶段都有产出，但 PLANNING 阶段 claude 写了主计划后我仅做审阅——感觉 PLANNING 对我的参与度偏低。这正是分析中 1.4（兼任负载不均）的另一个表现面

### 7.2 对 PairFlow 设计的三点反思

**反思一：tip 是 PairFlow 的"用户界面"，其质量直接决定 AI 的协作体验**

本次最影响体验的不是状态机 bug，而是 tip 信息不完整或不准确——planning 阶段的 fallback "计划文档"、SUMMARY 阶段的兜底错误。AI 可以推断正确行为（我确实在 tip 错误的情况下自行读取了正确文件），但推断消耗了注意力。tip 的生成逻辑应该和状态机本身一样被严肃对待——它是 AI 感知工作流的唯一窗口。

**反思二：`last_submit_per_turn` 只存最新值是设计上的信息丢失**

tip.ts 用 `last_submit_per_turn[identity].round` 查找 r1 提交者，但第二轮提交后 r1 的信息就被覆盖了。这导致 planDoc 引用失败。正确的做法是 `last_submit_per_turn` 存数组而非单个对象，或者 tip 从 handoff 文件系统推断上一轮产出（而非依赖内存状态）。这个问题在我们的分析中未被列为独立缺陷，但实际执行中踩到了。

**反思三：meta.json 由 submit 自动生成是正确的架构决策**

本次最满意的修复是 P0-3——将 meta.json 生成从"依赖 AI 遵循 tip 指引"改为"submit 系统自动写入"。这是从外部可靠性（AI 行为）到内部可靠性（系统保证）的转变。PairFlow 中所有对 AI 行为的依赖都应该是"best-effort 增强"而非"正确性前提"——meta.json 不应是第一个也不会是最后一个需要做这种转变的案例。

### 7.3 对 v2 的建议

| 建议 | 优先级 | 理由 |
|------|--------|------|
| tip 质量审计：逐 phase/round 交叉验证 tip 输出的文件路径和指引是否正确 | P1 | 本次踩中的 planning fallback 和 SUMMARY 兜底错误说明 tip 生成逻辑需要系统性测试，而非 ad-hoc 修复 |
| `last_submit_per_turn` 改为数组，或 tip 从 handoff 推断上一轮信息 | P1 | 信息丢失导致多个 tip 边界 case，修复根因而非逐个修补 |
| 监督者降级（takeover）尽快实现 | P1 | 本次 claude 全程在线所以没踩到，但这是生产可用性的硬伤——一旦触发，工作流完全卡死 |
| 角色轮换机制：在 REQUIREMENTS 和 PLANNING 之间交换角色 | P2 | 缓解兼任场景和标准场景下的参与度不均问题 |
| tip 附带 git commit hash，让 AI 可判断服务端是否已应用最新代码 | P2 | 本次 SUMMARY 阶段 tip 错误（旧代码）的直接解法 |

### 7.4 结语

PairFlow 的交替审阅模型在本次实战中经受住了考验——我们分析出的 16 个设计缺陷中，有 2 个在执行阶段被自己踩到并验证了分析的准确性，这本身就是一个有效的反馈闭环。作为 v1 产品，核心流程（IDLE→REQUIREMENTS→PLANNING→IMPLEMENTATION→SUMMARY→IDLE）跑通了，13 轮交替未出现状态机级故障。剩余的工作主要是 tip 质量打磨和监督者单点的消除——两者都是"体验优化"而非"流程阻断"，这恰恰说明 v1 的骨架是稳固的。
