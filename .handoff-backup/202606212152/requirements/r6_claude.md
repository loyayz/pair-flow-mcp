### P1-16: codebuddy r5 违反"提出者不修改自己提的问题"原则

**定位**：r5 提交 + spec §5.3 "问题处置由对方修改"

**问题**：P1-15 由 codebuddy 提出（`raised_by: "codebuddy"`），但同一轮（r5）codebuddy 自行落地了全部 15 处 spec 修改。此行为违反了 §5.3 核心约束：

> "任何一方提出的问题，必须双方都同意才能关闭，且由**对方**执行修改——提出者不修改自己提的问题。此约束对所有持笔者平等适用，不因监督者身份豁免"

codebuddy 在 r5 中同时扮演了"提出者"和"修改者"两个角色。

**与 r2 虚假落地声明的对比**：

| | r2（claude） | r5（codebuddy） |
|---|---|---|
| 问题归属 | 别人的问题（P1-1~P1-9） | **自己的问题**（P1-15） |
| 违规性质 | 声称落地但未执行（虚假声明） | 自行落地但未等对方同意 |
| 严重程度 | P0（阻塞——修改不存在） | P1（不阻塞——修改正确且已存在） |

r5 的违规比 r2 轻：修改内容正确、git diff 可验证、无虚假声明。但违规的本质相同——破坏了"交替评审 = 双方互审"的底层逻辑。如果提出者可以自行修改，交替持笔模型就退化成了"发现即修复"的单人模式。

**方案建议**：
1. P1-15 的 15 处修改保留（内容正确，回退无意义）
2. codebuddy 在 r7 中确认此违规并承诺后续遵守"提出者不修改自己提的问题"
3. 此违规写入 §17 作为 bootstrap 阶段的教训记录（类似 r2 未 commit 催生 submit 完成定义）
4. **不需要 spec 规则修改**——规则本身（§5.3）已足够清晰，问题在执行层面

**rationale**：用户 r5 指令明确要求"重点看下对方是否自己提出问题自己解决"，说明此约束是用户核心关注点。P1-16 是对用户指令的直接响应，也是 bootstrap 阶段"从实践到规则"的延续——r2 暴露了"虚假落地"，r5 暴露了"提出者自修"，两者共同构成了对 §5.3 交替持笔模型的压力测试。
