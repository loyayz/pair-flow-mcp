## P1-34: §3 handoff 目录结构图未包含盲审产出文件

## P1-35: §5.3 "三分判断"表未覆盖盲审后发现 issue 的情况

**定位**：§5.3 line 339-343

**问题**：advance 判断表定义三种情况：① 干净收敛 ② 遗留强关 ③ 有 P0。但盲审后可能产生 **P1（非阻塞）issue**——这些不属于 P0（情况 ③），也不属于"干净收敛"（情况 ① 条件为"无 open P0 + 无被强关的僵持 issue"，未提 P1）。盲审发现的 P1 应如何处理？

若有 open P1（盲审发现），按"无 open P0 → 干净收敛 → advance"的当前逻辑，P1 会被直接带过。但盲审初衷是"发现所有遗漏问题"，P1 也应处理完成后再 advance。

**方案建议**：情况 ① 条件改为"无 open issue（P0+P1+P2）"，或新增情况 ①-a"有 P1——可 advance 也可继续讨论（监督者判断）"。

## P1-37: §12 REQUIREMENTS 初始化 last_submit_per_turn 字段与 §5.1 schema 不一致

**定位**：§12 line 793 vs §5.1 line 175-184

**问题**：§12 REQUIREMENTS 初始化 `last_submit_per_turn` 含 `round` 和 `sub_phase` 字段，但 §5.1 schema 未声明这两个字段。这与 r18 P1-25 是同根问题——§12 和 §5.1 的 last_submit_per_turn 定义不同步。

但本问题关注的是**REQUIREMENTS 阶段的初始化值**：非 IMPLEMENTATION 阶段 `round` 初始化为 `null`，收敛检查不依赖 round 匹配（§7："需求/计划阶段不依赖 round 匹配"）。那为什么 REQUIREMENTS 还要初始化 round 字段？如果 round 对需求阶段无意义，初始化它为 null 是一种浪费。

**方案建议**：要么从非 IMPLEMENTATION 阶段的 last_submit_per_turn 中移除 round 和 sub_phase（仅 IMPLEMENTATION 使用），要么在 §5.1 schema 中补全并在 §12 中标注"非 IMPLEMENTATION 阶段为 null"。

## 盲审反思

1. **独立性损失**：用户分享了 r18 后我才执行盲审，真正的"盲"已不成立。bootstrap 阶段无 PairFlow 强制隔离，独立性只能靠自觉——而这正是 P1-22 指出的问题。**本轮盲审的独立性声明是诚实但不完整的。**

2. **盲审的真实价值**：即使独立性不完整，我仍找到了 5 个 r18 未覆盖的问题。这说明盲审不是"两个人各找一遍，取并集"——是**不同视角看到不同问题**。即使我知道 codebuddy 发现了 11 个，我的视角仍然补了 5 个不同的。

3. **盲审机制自身的不完整性**：r18 和 r19 合计发现 16 个问题，其中至少 6 个（P1-27/28/30/31/32 + P1-39）是盲审机制的集成缺陷。这证明 P0-3 的初始落地是概念级而非集成级——写了机制定义，没做全 spec 对齐。
